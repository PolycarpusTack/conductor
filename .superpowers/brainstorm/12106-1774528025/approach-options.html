Agent Creation: How Deep Should We Go?

Three approaches — from practical enrichment to full lifecycle platform. Click the one that matches your vision.

A

Enriched Agent Profile (Recommended)

Add role, capabilities, autonomy level, and max concurrent tasks to the existing agent model. The creation dialog becomes a richer form but stays a single step. Takes your Stage 1 (Identity) concepts and makes them practical fields in the current system.

What you get from your framework:
Stage 1: Persona (role, mission) + Guardrails (autonomy level, task limits)
Stage 3: Capabilities as tags (what tools/skills the agent has)

Schema change: 4 new fields on Agent model
Effort: ~2 hours. Ships today.
Tradeoff: No versioning, no eval harness, no ETL pipelines
B

Multi-Step Agent Wizard

A tabbed/stepped creation flow walking through Identity, Capabilities, Orchestration, and Deployment config. Each step maps to one of your framework stages. The agent form becomes a mini-wizard instead of a single dialog.

What you get from your framework:
Stage 1: Identity tab (name, persona, security level)
Stage 3: Capabilities tab (tool catalog, scoping, HITL toggles)
Stage 4: Orchestration tab (architecture choice, fallback logic)
Stage 5: Basic deployment config (environment selector)

Schema change: ~8 new fields + JSON config column
Effort: ~1 day. Needs new UI component.
Tradeoff: More complex form, fields won't have real backend logic yet (tool catalog is metadata only, no actual MCP integration)
C

Full Lifecycle Platform (Your 2026 Vision)

Build all 6 stages: Identity + ETL pipelines + MCP tool governance + Orchestration engine + CI/CD with eval harness + Live observability dashboard. Includes the evaluation report and live monitor from your HTML mockups.

What you get from your framework:
All 6 stages fully implemented
Pre-deployment eval report (your Stage 5 design)
Live production monitor (your Stage 6 design)

Schema change: 5+ new models, config tables, eval results
Effort: Weeks to months. Major architectural shift.
Tradeoff: AgentBoard becomes a different product. Most fields have no backing infrastructure yet. Risk of building UI for systems that don't exist.

My Recommendation: Start with A, design toward B

Option A gives you real, usable fields today that make agents more than just a name and emoji. The schema additions are designed so that Option B's wizard can wrap around them later without migration pain. Option C is the north star but needs infrastructure (eval pipelines, OTel, MCP runtime) that's beyond the current codebase.
